Explain how a HashMap works internally.

A HashMap uses an array of buckets. When you put a key-value pair, it:
1. Computes hashCode() of the key
2. Applies hash() to spread bits (reduce collisions)
3. Computes index = (n-1) & hash
4. Stores in the bucket. If collision, uses linked list (or tree if >8 entries)

Time Complexity: O(1) average for get/put, O(n) worst case (all collide)
Load factor: 0.75 default — resize when 75% full

What's the difference between REST and GraphQL?

REST: Multiple endpoints, fixed response structure, over-fetching/under-fetching common, stateless, cache-friendly.

GraphQL: Single endpoint, client specifies exact fields needed, no over-fetching, strong typing via schema, harder to cache at HTTP level.

When to use: REST for simple CRUD, GraphQL for complex data relationships or mobile apps with varying data needs.

Design a URL shortener like Bitly.

Requirements: Shorten URLs, redirect, custom aliases, analytics, 1M+ requests/day.

API: POST /shorten {url, custom_alias?} → {short_url}
GET /{short_code} → 302 redirect

Database: URL mapping table (short_code → original_url, created_at, clicks). Use base62 encoding for short codes. 7 chars = 62^7 ≈ 3.5 trillion unique URLs.

Cache: Redis for hot URLs (80% of traffic is 20% of URLs).

Analytics: Kafka → Click stream → aggregate hourly.

Design a rate limiter.

Algorithms:
• Token Bucket — tokens refill at constant rate, burst allowed
• Leaky Bucket — requests processed at fixed rate, queue for bursts
• Sliding Window — most accurate, counts requests in last N seconds

Implementation: Redis with INCR and EXPIRE. Key = user_id:endpoint, TTL = window size.

Headers: X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset
